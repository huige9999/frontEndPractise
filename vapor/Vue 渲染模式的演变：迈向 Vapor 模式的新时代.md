## 为什么Vue的渲染模式会演变？

### 基石：响应式编程带来的开发优势
Vue 是一个响应式框架,这意味着我们可以声明式地将状态（数据）映射到视图（DOM），当状态变更时，框架负责自动、高效地更新视图。这极大地简化了 UI 开发，避免了手动、繁琐且易错的 DOM 操作。

![alt text](<image/Vue 渲染模式的演变：迈向 Vapor 模式的新时代/image-1.png>)

### 核心挑战：DOM更新的性能瓶颈
然而，“如何高效地更新视图” 是所有现代前端框架面临的核心挑战。直接操作真实 DOM 的成本很高（涉及重排、重绘），频繁或不必要的 DOM 操作是性能瓶颈的主要来源。因此，一个 “高效的渲染机制” 对于保证应用流畅度（Frame Rate）和用户体验至关重要。这就是 Vue 不断迭代其渲染策略的根本原因。


## 渲染模式的演进：一部追求极致性能的优化史

Vue渲染模式的演变：
- 1.0：DOM-based templating（基于DOM的模板）
- 2.0：Pure Virtual DOM（纯虚拟DOM）
- 3.0：Compiler-enhanced Virtual DOM（编译器增强型虚拟DOM）
- 3.6：Vapor Mode（蒸汽模式）


## Vue 1.0：基于真实 DOM 的模板

### 核心理念：贴近原生，直接操作
- “所见即所得”的模板:
 Vue 1.0 直接将 HTML 文件中的标准 DOM 结构作为模板来源。开发者编写的就是最终运行的（部分）DOM 结构。

- 工作方式:

    1. 挂载: Vue 实例 (new Vue({ el: '#app' })) 附加到页面上已存在的 DOM 元素 (#app)。

    2. 编译: Vue 遍历这个真实 DOM 树，查找 v- 指令（如 v-on, {{ }}）。

    3. 绑定: 为每个指令建立数据 (data) 与对应 DOM 元素之间的直接响应式联系（通过 Watcher）。

    4. 更新: 数据变更时，对应的 Watcher 直接触发真实 DOM 操作来更新视图。

### 关键特征
- 模板即 DOM: 模板信息源于运行时实际的 DOM 结构。

- 指令驱动: v-* 指令为静态 DOM 赋予动态行为。

- 细粒度更新: 数据变更理论上只触发直接相关的 DOM 更新。

- 运行时编译: 模板解析和绑定在浏览器中实时进行。


### 主要弊端：性能瓶颈与局限性

- 性能问题:

    - 启动慢: 遍历真实 DOM 并解析指令，初始编译成本高。

    - 内存/计算开销大: 大量 Watcher 实例占用内存，依赖追踪计算复杂。

    - DOM 操作昂贵: 直接且可能频繁地操作真实 DOM，容易导致性能下降（重排/重绘）。

- 优化与维护困难:

    - 难以静态优化: 运行时才能确定模板结构，限制了编译时优化的可能性。

    - 模板耦合: 复杂逻辑易混入 HTML，降低可读性、可维护性。

- 扩展性限制:

    - SSR 复杂: 服务端模拟浏览器 DOM 环境成本高、效率低。
    - 跨平台困难: 强依赖浏览器 DOM API，难以拓展到非浏览器环境（如 Native）

### 小结
Vue 1.0 的模式虽然直观，但其对真实 DOM 的依赖和运行时编译带来了显著的性能瓶颈和多方面的局限性，这直接催生了后续版本对渲染机制的重大革新（如：接下来要介绍的虚拟 DOM）。
